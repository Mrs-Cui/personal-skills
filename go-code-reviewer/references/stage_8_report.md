# 阶段8：汇总报告生成

## 目标
聚合阶段2-7的所有发现，去重、排序后生成结构化审核报告。

## 输入
- 阶段1的上下文摘要（审核模式、范围、规则集、严格度分布）
- 阶段2-7的所有发现（问题列表）

---

## 处理流程

### 1. 收集

收集阶段2-7的所有问题，每个问题必须包含：
- 严重程度（🔴严重 / 🟡警告 / 🔵建议）
- 问题ID（如 `bug/nil-pointer`）
- 文件位置（`file:line`）
- 所属阶段
- 代码片段
- 问题描述
- 修复建议

### 2. 去重

同一位置（同文件同行号或同一代码块）被多个阶段标记的问题，合并为一条：

- 保留最高严重程度
- 注明涉及的多个维度
- 合并问题描述和修复建议，保留各阶段的关键信息

**示例：**
阶段3发现 goroutine 泄漏（`bug/goroutine-leak`，🟡警告），阶段5发现同一位置可被外部利用（`security/goroutine-dos`，🔴严重）→ 合并为一条：
- 严重程度取 🔴严重
- 问题ID取更高严重程度的 `security/goroutine-dos`
- 问题描述注明"涉及维度：潜在bug、安全漏洞"
- 修复建议合并两个阶段的方案

### 3. 排序

按以下优先级排序：
1. 严重程度降序：🔴严重 > 🟡警告 > 🔵建议
2. 同级别按文件路径字母序
3. 同文件按行号升序

### 4. 生成结构化摘要

加载 `references/output_format.md` 中的结构化摘要模板，填充：
- 审核模式、审核范围、使用规则集、严格度分布（来自阶段1）
- 严重程度统计表（去重后重新计数）
- 按维度分布统计（去重后重新计数；合并的问题计入主问题ID所属维度）

### 5. 嵌入架构分析图表

**[必须]** 加载 `references/output_format.md` 中的"架构分析图表"模板，将阶段1中 go-code-analyzer 产出的架构分析以 Mermaid 图表形式嵌入报告。位于结构化摘要之后、问题列表之前。

必须包含以下 5 个图表：
1. **入口点列表**（表格）— 所有新增或修改的入口点
2. **函数调用层级图**（Mermaid graph TD）— 从入口层到DAO层的完整调用链，按层分 subgraph
3. **服务依赖图**（Mermaid graph LR）— 标注调用方式和关键参数
4. **核心业务流程图**（Mermaid flowchart TD）— 每个关键入口点一张，包含所有分支
5. **数据库表关系图**（Mermaid erDiagram）— 仅包含本次变更涉及的表

图表的具体格式要求参见 `references/output_format.md` 中的"架构分析图表"章节。

### 6. 生成逐条详细评论

加载 `references/output_format.md` 中的逐条评论模板，按排序后的顺序逐条输出。合并的问题在阶段字段注明多个阶段来源。

### 7. 生成末尾总结

- **整体评价**：一句话概括代码质量
- **最需关注的 Top 3 问题**：按严重程度取前3；不足3个则用低级别补充
- **需人工确认项**：汇总所有带 `[业务意图不明确]` 标记的问题；无则省略该小节
- **无问题时**：使用 `references/output_format.md` 中的简化格式

---

## 输出格式

引用 `references/output_format.md` 中定义的完整格式。本文件不重复定义格式，仅描述生成逻辑。

---

## 质量要求

1. **问题描述必须说明根因和影响**，不能只描述现象
2. **修复建议必须具体可操作**，尽可能给出代码示例
3. **去重后不能丢失任何阶段的关键信息**
4. **统计数字必须准确**——基于去重后的问题列表重新计数，不能简单累加各阶段数字
