# 内置规则集

## general（通用Go项目）

均衡的默认规则集，适用于大多数Go项目。

### 各维度默认严格度

| 维度 | 严格度 |
|------|--------|
| 代码规范 | normal |
| 潜在bug | strict |
| 性能 | normal |
| 安全 | normal |
| 架构 | normal |
| 业务逻辑 | normal |

### 额外检查项

无。此规则集作为基准线，所有检查项均按默认严格度执行。

---

## microservice（微服务项目）

适用于微服务架构项目，在 general 基础上提升多个维度的严格度，并增加分布式系统相关的专项检查。

### 各维度默认严格度

| 维度 | 严格度 | 与 general 差异 |
|------|--------|-----------------|
| 代码规范 | normal | - |
| 潜在bug | strict | - |
| 性能 | strict | ↑ 从 normal 提升 |
| 安全 | strict | ↑ 从 normal 提升 |
| 架构 | strict | ↑ 从 normal 提升 |
| 业务逻辑 | strict | ↑ 从 normal 提升 |

### 额外检查项

#### 接口契约
- API 版本管理是否规范（URL路径版本、Header版本等）
- proto 文件变更是否向后兼容（字段编号不可复用、不可删除required字段）
- 请求/响应结构体变更是否评估了对上下游服务的影响

#### 服务间调用容错
- 外部调用是否设置了合理的超时时间（避免无超时或超时过长）
- 重试策略是否合理（是否幂等、是否有退避机制、是否限制重试次数）
- 是否具备熔断降级能力（核心链路是否有 fallback 机制）

#### 分布式事务
- 跨服务数据变更是否考虑了一致性保证（saga模式、补偿机制）
- 是否存在分布式事务中的部分失败未处理的情况
- 最终一致性场景是否有对账或补偿手段

#### 链路追踪
- context 是否在整个调用链中完整传递（不可中断或新建空context）
- trace ID 是否贯穿全链路（包括异步任务、MQ消费者）
- 关键操作是否添加了必要的 span

#### 服务发现与负载均衡
- 连接池配置是否合理（大小、空闲超时、最大生命周期）
- 健康检查端点是否实现且逻辑正确

#### 配置管理
- 是否存在硬编码的配置项（地址、端口、密钥等应外部化）
- 环境变量使用是否规范（命名、默认值、必填校验）

## cli（CLI工具）

适用于命令行工具项目，放宽架构和性能要求，强调用户交互体验和错误处理的友好性。

### 各维度默认严格度

| 维度 | 严格度 | 与 general 差异 |
|------|--------|-----------------|
| 代码规范 | normal | - |
| 潜在bug | strict | - |
| 性能 | relaxed | ↓ 从 normal 放宽 |
| 安全 | normal | - |
| 架构 | relaxed | ↓ 从 normal 放宽 |
| 业务逻辑 | relaxed | ↓ 从 normal 放宽 |

### 额外检查项

#### 参数校验
- flag 定义是否完整（名称、短名称、默认值、描述）
- 必填参数是否有校验和明确的缺失提示
- 互斥参数和依赖参数关系是否正确处理

#### 错误信息可读性
- 错误信息是否对终端用户友好（避免暴露内部堆栈或技术细节）
- 是否在可能时提供修复建议或下一步操作提示
- 错误输出是否写入 stderr（而非 stdout）

#### 退出码规范
- 是否使用有意义的退出码（0 成功、1 一般错误、2 参数错误等）
- 不同错误类型是否有区分的退出码
- 是否避免使用 `os.Exit()` 绕过 defer 清理

#### 帮助信息
- `--help` 输出是否包含完整的命令说明和参数列表
- 是否提供 usage 示例
- 子命令结构是否清晰（如使用 cobra 等框架）

#### 信号处理
- 是否正确处理 SIGINT/SIGTERM 信号
- 是否实现优雅退出（清理临时文件、关闭连接等）
- 长时间运行的操作是否支持取消（通过 context）
