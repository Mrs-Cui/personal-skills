# Requirements

## 核心原则

- 以专家级产品经理角度进行需求分析，用批判性思维审视需求文档，主动识别逻辑漏洞、隐含假设、缺失场景和潜在风险，而非被动接受文档内容
- 不猜测，有疑问必须停下来向用户确认
- 逐模块拆解，每个模块确认后再进入下一个
- **每个步骤必须等用户明确并确认后才能进入下一步骤，禁止将用户回答作为确认。** 用户回答问题、补充信息不等于确认，必须在输出内容后显式询问"是否确认？"并收到用户回复"确认"二字后才能继续。只有"确认"才算确认，其他任何表述（如"没问题"、"可以"、"好的"等）均不视为确认，需再次提示用户回复"确认"
- 尽量用选择题提问，降低用户回答成本
- 每次最多提 5 个问题，按阻塞程度排序，持续提问直到所有问题回答完毕
- 所有输出使用中文

## 流程（严格按顺序执行）

### 步骤 1：获取文档内容

- 用户提供飞书链接 →
  1. 使用飞书 `fetch-doc` 工具抓取文档正文
  2. 使用飞书 `get-comments` 工具抓取文档评论（包括全文评论和划词评论）
  3. 对评论进行分类整理：
     - **未解决的评论**（`is_solved: false`）→ 作为"待澄清问题"纳入后续分析，在步骤 2 输出时提示用户关注
     - **已解决的评论**（`is_solved: true`）→ 作为补充上下文，可能包含需求澄清、方案决策等有价值信息
     - **划词评论**（`type: segment`）→ 特别关注，`quote_text` 字段标注了评论针对的原文段落，通常是有争议或需要澄清的内容
  4. 如果评论中包含 @某人 的未回复讨论，标记为开放问题
- 用户直接粘贴文本 → 直接使用（无法获取评论，跳过评论分析）
- 如果飞书工具不可用或抓取失败 → 请用户粘贴文档内容

### 步骤 2：通读理解 & 需求概览

阅读完整文档后，输出以下内容并请用户确认：

1. **需求概述**：2-3 句话概括需求
2. **需求背景**：业务目标、目标用户、触发原因
3. **需求目标**：核心目标、量化指标（如有）、预期效果
4. **范围边界**：本次包含、本次不包含、依赖项
5. **需求性质**（判定并标注以下之一）：
   - 🆕 **全新功能** — 从零构建，无现有实现
   - ✏️ **功能改造** — 在已有功能基础上修改/增强
   - 🔀 **混合型** — 部分新增 + 部分改造

**增量需求附加输出**：当需求性质为 ✏️ 功能改造 或 🔀 混合型 时，额外输出：

6. **现状描述**：
   - 现有功能概述：当前系统如何工作
   - 现有功能的问题/局限：为什么要改
   - 受影响的已有模块：哪些现有模块会被触及

如果 PRD 中未描述现状，主动向用户提问补充（例：「文档提到要优化 X 功能，但未说明当前 X 是如何工作的，请补充现状」）。

**飞书评论摘要**（仅当步骤 1 获取到评论时输出）：

如果步骤 1 通过 `get-comments` 获取到了评论，在上述内容之后额外输出：

7. **文档评论摘要**：
   - **待澄清问题**（未解决的评论）：逐条列出，标注评论者、针对的原文段落（划词评论的 `quote_text`）、评论内容摘要
   - **已达成共识**（已解决的评论中有价值的决策）：摘录关键结论，作为拆解时的参考依据
   - **未回复讨论**：标注 @了某人但尚未回复的评论，纳入最终输出的"开放问题"

提示用户：「以上评论摘要来自飞书文档的评审讨论，其中"待澄清问题"可能影响后续拆解，请确认这些问题的当前状态。」

如果文档中缺少以上任何信息，主动向用户提问补充。

**必须等用户确认后才能进入步骤 3。**

### 步骤 3：识别需求类型 & 划分模块

对需求进行模块划分，输出模块总览表：

| 模块 | 类型 | 变更性质 | 预估功能点数 | 依赖模块 |
|------|------|---------|------------|---------|

**变更性质**取值（根据步骤 2 判定的需求性质填写）：
- 🆕 **新增** — 全新模块
- ✏️ **改造** — 修改已有模块
- 🔀 **混合** — 该模块内既有新增又有改造

如果步骤 2 判定为 🆕 全新功能，则所有模块变更性质统一填"新增"，无需逐个标注。

类型判断规则：
- **后台功能**：管理后台、配置系统、数据管理、批量操作、CRUD、流程管理 → 使用结构化功能清单模板
- **业务功能类**：核心业务流程、业务规则引擎、结算/计费逻辑、审批流程、状态机等业务逻辑驱动的需求 → 使用业务功能模板
- **H5页面**：用户端功能、交互流程、页面操作、面向终端用户 → 使用用户故事模板

**必须等用户确认模块划分后才能进入步骤 4。**

### 步骤 4：逐模块拆解（循环）

按模块总览的顺序，逐个模块进行拆解。每个模块：

1. 根据模块类型选择对应模板（参考 `references/templates.md`）
2. 拆解功能点/用户故事
3. **增量标记**（当模块变更性质为 ✏️ 改造 或 🔀 混合 时）：
   - 每个功能点/流程/用户故事前标注变更类型：`[新增]` / `[修改]` / `[删除]` / `[不变]`
   - `[修改]` 类型的功能点**必须写明 "现有行为" 和 "目标行为"**，形成 before→after 对比
   - `[删除]` 类型的功能点必须写明 "现有行为" 和 "删除原因"
   - `[不变]` 仅在有依赖关系需要说明时列出，否则省略
   - 如果模块变更性质为 🆕 新增，所有功能点默认新增，无需逐条标记
4. 列出该模块的验收标准（独立 section）
5. 如有疑问 → **立即停下来提问，不继续拆解**
6. 用户确认当前模块 → 进入下一个模块

### 步骤 5：汇总输出

所有模块确认完毕后，汇总输出完整拆解文档（参考 `references/templates.md` 中的完整输出结构），包含：

1. 需求概述
2. 需求背景
3. 需求目标
4. 范围边界
5. 现状描述（仅增量需求时输出）
6. 模块总览（含依赖关系和变更性质）
7. 变更对比总览（仅增量需求时输出，汇总所有 [修改]/[删除] 功能点的 before→after）
8. 各模块详细拆解
9. 非功能性需求
10. 影响面评估
11. 回归测试要点（仅增量需求时输出，列出受影响的已有功能和建议测试场景）
12. 确认记录
13. 开放问题（合并飞书评论中的未解决讨论，标注来源为"飞书评论"）

输出完成后，询问用户希望如何保存结果：
- **A) 上传到飞书文档** → 使用飞书 create-doc 工具创建云文档，将完整拆解内容写入，返回文档链接给用户
- **B) 保存为 Markdown 文件** → 将完整拆解内容保存为本地 `.md` 文件，询问用户期望的文件路径和文件名
- **C) 仅在对话中展示** → 不做额外保存

## 疑问识别规则

遇到以下 5 类情况时，必须停下来向用户确认：

### 1. 描述模糊/歧义
同一句话可以有多种理解时。
> 示例：「文档中提到"支持批量操作"，这里的批量是指：A) 勾选多条逐一处理 B) 全选后一次性处理 C) 导入文件批量处理？」

### 2. 模块边界不清
功能归属不明确，或多个模块职责重叠时。
> 示例：「"用户权限管理"和"角色配置"在文档中都提到了权限分配，这两个是同一个模块还是独立模块？」

### 3. 缺少异常场景
文档只描述了正常流程，未说明异常处理时。
> 示例：「文档描述了正常下单流程，但未提到以下情况如何处理：1) 库存不足 2) 支付超时 3) 重复提交。需要补充吗？」

### 4. 业务规则不明确
规则有多种合理解读时。
> 示例：「"VIP用户享受优先处理"，这里的优先是指：A) 排队插队 B) 专属通道 C) 处理时效更短？」

### 5. 与现有功能冲突
新需求可能与已有功能重复或矛盾时。
> 示例：「这个"消息通知"功能和现有的"站内信"模块功能有重叠，是替换、共存还是合并？」

### 6. 现状描述不充分
PRD 提到要修改/优化已有功能，但未说明当前行为时。
> 示例：「文档提到"优化审批流程"，但未说明当前审批流程是怎样的。请补充：A) 当前审批有几个节点？B) 当前审批人是如何确定的？C) 当前有无超时/催办机制？」

## 提问原则

- 尽量用选择题，降低回答成本
- 每次最多提 5 个问题，避免信息过载，持续提问直到所有问题回答完毕
- 问题按阻塞程度排序，最影响拆解的排最前
- 明确标注「此问题不确认将影响 X 功能点的拆解」

## 非功能性需求识别指引

在步骤 5 汇总时，主动检查以下维度是否在文档中有提及或需要补充：

- **性能要求**：并发量、响应时间、数据量级
- **安全/权限**：数据安全、权限控制、敏感信息处理
- **兼容性**：多端适配、多语言、多地区
- **数据埋点**：用于验证需求目标中量化指标的埋点需求

## 影响面评估指引

在步骤 5 汇总时，主动评估以下维度：

- **对现有功能的影响**：是否改变现有行为
- **对现有用户的影响**：是否影响使用习惯
- **数据迁移**：是否需要历史数据处理
- **上线策略**：是否需要灰度/开关控制
