# Go Code Analyzer

分析 Go 代码库，提取业务逻辑、生成 Mermaid 流程图、构建函数调用层次和服务依赖关系文档。

## 分析工作流

### Step 1: 获取项目上下文

- 读取项目 `CLAUDE.md` 了解架构分层规则和目录结构
- 检查 `knowledge/` 目录是否已有目标模块的知识文件，避免重复分析
- 确认项目的实际分层命名（如 `iface → app → services → dao → conn`），不要用默认假设

### Step 2: 代码发现与入口点识别

根据用户需求确定分析范围：
- **Git diff**: `git diff --name-only` 获取变更文件列表
- **指定目录/文件**: 直接分析目标路径
- **指定函数**: 从函数开始向下追踪

扫描入口点类型：
- HTTP 路由：搜索 `router.go` 或路由注册代码
- MQ 消费者：搜索 `consumer/` 目录下的 handler 注册
- 定时任务：搜索 `schedule/` 目录下的 cron 注册
- gRPC 方法：搜索 proto 实现方法

### Step 3: 构建调用层次（渐进式）

**渐进式分析，避免上下文溢出：**

1. 用 `get_symbols_overview` 获取目标文件符号概览（不读代码体）
2. 用 `find_symbol` + `depth=1` 获取入口函数的一级调用关系
3. 对每个被调用函数，按项目分层规则分类层级
4. 只对关键路径函数读取 body（`find_symbol` + `include_body=true`）
5. 用 `find_referencing_symbols` 追踪跨包调用关系
6. **超过 30 个函数时分批**：先完成主路径，再按需分析分支

**每个函数记录**：函数签名（完整参数+返回值）、层级、职责（一句话）、调用的子函数、调用模式（同步/异步/并行）

**标记特殊情况**：递归调用、goroutine、sync.WaitGroup/errgroup、条件分支

### Step 4: 提取服务依赖

| 依赖类型 | 需记录 |
|----------|--------|
| HTTP/gRPC | 服务名、接口、方法、超时、重试、失败处理 |
| 消息队列 | 类型(Kafka/RabbitMQ)、topic、消息格式、失败处理 |
| 数据库 | 类型、表名、操作(CRUD)、事务边界 |
| 缓存 | 类型、key 模式、TTL、失败降级 |

### Step 5: 选择输出模式

| 模式 | 触发条件 | 输出内容 |
|------|----------|----------|
| **快速** | 单个函数/小范围 | 调用链列表 + 直接依赖 |
| **标准** | 模块/目录级别 | 调用层次图 + 依赖图 + 入口点 + 核心流程 |
| **完整** | 服务级/PR 全量分析 | 完整文档（概述、调用层次、依赖、入口点、流程图、决策逻辑、数据流、错误处理、待确认项） |

模板和示例见 [references/EXAMPLES.md](references/EXAMPLES.md)。

## 工具策略

**优先使用语义工具，禁止手工解析代码：**

| 目的 | 工具 | 说明 |
|------|------|------|
| 文件符号概览 | `get_symbols_overview` | 了解文件有哪些类/函数/方法 |
| 函数签名 | `find_symbol` + `include_info=true` | 完整参数和返回值类型 |
| 函数体 | `find_symbol` + `include_body=true` | 仅在需要理解具体逻辑时用 |
| 子函数列表 | `find_symbol` + `depth=1` | 获取结构体所有方法 |
| 调用追踪 | `find_referencing_symbols` | 谁调用了这个函数 |
| LSP 调用层次 | LSP `incomingCalls`/`outgoingCalls` | 精确调用树 |
| 模式搜索 | `search_for_pattern` | 路由注册、topic 定义等 |
| 目录浏览 | `list_dir` | 模块结构 |

**禁止**：不要用 `cat`/`grep` 手工解析 Go 文件，不要一次性读取大文件全文。

## Mermaid 图规范

- **调用层次图**: `graph TD`，入口蓝 `#e1f5ff`、服务红 `#ffe1e1`、DAO 绿 `#e1ffe1`、外部调用黄 `#fff4e1`、异步紫 `#f0e1ff`
- **依赖关系图**: `graph LR`，标注调用方式（HTTP/gRPC/Kafka/Redis/MySQL）
- **详细流程图**: `flowchart TD`，含所有分支和错误处理
- **节点上限**: 单张 ≤ 50 节点；超出按入口点拆分为独立子图

## 不确定性处理

不明确时**不要推测**，用标签标记并附带代码位置（file:line）、当前理解、具体问题：
- `[需确认]` — 业务逻辑不明确
- `[依赖不明确]` — 外部服务/接口信息不完整
- `[调用关系不明确]` — 调用路径不确定

## 输出自检

- [ ] 每条调用关系有源码依据（标注 file:line）
- [ ] 无推测性调用关系
- [ ] Mermaid 语法正确（无未闭合引号/括号）
- [ ] 单张图 ≤ 50 节点
- [ ] 函数签名完整（含参数类型和返回值）
- [ ] 不确定项均已标签标记

## 约束

**必须**: 所有函数调用含签名+层级+调用模式、所有外部依赖含接口+超时+失败处理、所有 DB/缓存操作。

**禁止**: 推测业务逻辑、省略错误处理分支、忽略异步/并发、遗漏外部依赖。
